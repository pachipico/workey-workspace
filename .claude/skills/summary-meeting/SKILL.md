# summary-meeting

WorKey 프로젝트 에이전트 회의 자료를 팀별 실행 작업 정리본으로 재구성하고 Obsidian vault에 저장하는 스킬.

회의 결정문서(`decision.md`)는 합의·매트릭스·KPI 형태로 압축되어 있어, 그대로는 각 팀이 무엇을 해야 할지 한눈에 보이지 않는다. 이 스킬은 그 압축된 결정을 **팀 관점으로 풀어 쓴** 실행 문서로 변환한다.

## 출력 구조 (디렉토리 기반)

이 스킬은 단일 파일이 아니라 **디렉토리 + 다중 파일** 구조로 저장한다:

```
10. Projects/WorKey/meeting-log/{YYYY-MM-DD}-{slug}/
├── sumup.md       # 통합 마일스톤 표 (액션 아이템 + 팀 간 미해결) + 한 줄 결론
├── pm.md          # PM 섹션 + PM owner 액션·미해결만 (회의 참여 시)
├── backend.md     # 백엔드 섹션 + 백엔드 owner 액션·미해결만
├── frontend.md    # 프론트엔드 섹션 + 프론트엔드 owner 액션·미해결만
├── design.md      # 디자인 섹션 + 디자인 owner 액션·미해결만
└── ux.md          # UX 섹션 + UX owner 액션·미해결만 (회의 참여 시)
```

이렇게 분리하는 이유:
- **`sumup.md`는 "전체 조망"용** — PM·team-lead가 마일스톤 추적, 신규 합류자가 회의 결과 한눈에 보기
- **`{team}.md`는 "작업 반영"용** — 각 팀이 자기 작업 문서(`docs/{team}/...`)에 결과를 반영할 때 클로드에게 던지는 컨텍스트가 작아진다
- 두 파일을 같이 두지 않으면 클로드 작업 시 sumup 전체를 넘기게 되어 컨텍스트가 폭발한다

## 두 파일의 역할 분리 (절대 섞지 말 것)

`sumup.md`와 `{team}.md`는 **다른 독자를 향한 다른 문서**이다:

- `sumup.md` — **마일스톤 추적용**. 팀별 풀어쓰기·시스템 그림·JSON 페이로드는 **포함하지 않는다**. 통합 액션 아이템 표·팀 간 의존성·한 줄 결론만 담아 분량을 슬림하게 유지한다
- `{team}.md` — **작업 반영용**. 시스템 그림·MUST·SHOULD·백로그·의존성·풀어쓰기 4가지(설명/JSON/목적/fallback)·표 보존이 모두 들어간다

`sumup.md`에 팀별 섹션을 넣으면 분리본이 redundant가 되고, `{team}.md`에 통합 액션 아이템 표를 통째로 복사하면 다른 팀 행까지 들어가 "내 작업"이 흐려진다. 두 파일이 자기 역할에 충실해야 한다.

## 트리거 예시

- `frontend-kickoff-meeting 회의 정리해줘`
- `kickoff 회의 요약해서 옵시디언에 저장해줘`
- `meeting/{안건}/ 정리`
- `이번 회의 팀별로 풀어줘`

`{안건}`은 `meeting/` 하위 디렉토리명이다 (예: `frontend-kickoff-meeting`).

---

## 입력 파일

`meeting/{안건}/` 하위에서 다음을 모두 읽는다:

- `agenda.md` — 회의 안건 (있으면)
- **round 파일 전체** — `evidence/*-round*.md`, `rebuttals/*-round*.md` 둘 다 포함. 라운드 수에 제한을 두지 말고 발견되는 모든 round 파일을 수집 (round1, round2, round3, … round10이라도 끝까지)
- `decision.md` — **필수**. 없으면 작업 중단하고 사용자에게 알린다
- `minutes.md` — 회의록 (있으면)

회의 디렉토리 구조는 회의마다 다를 수 있다. `evidence/`·`rebuttals/`가 있을 수도, 회의 디렉토리 직속에 round 파일이 있을 수도 있다. 둘 다 글롭으로 수집한다.

### 참여 팀 동적 감지

round 파일명의 접두사로 회의 참여 팀을 감지한다:

| prefix | 팀 | 출력 파일명 |
|---|---|---|
| `pm-round*.md` | 기획(PM) | `pm.md` |
| `backend-round*.md` | 백엔드 | `backend.md` |
| `frontend-round*.md` | 프론트엔드 | `frontend.md` |
| `design-round*.md` | 디자인 | `design.md` |
| `ux-round*.md` | UX | `ux.md` |

**round 파일이 없는 팀은 출력 파일 자체를 생략한다.** 회의에 참여하지 않은 팀의 빈 분리본을 만들지 말 것. `sumup.md` 안의 통합 액션 아이템·미해결 항목 표에서도 그 팀의 행이 없을 가능성이 높으니 자연스럽게 누락된다.

---

## 절대 금지 사항

다음 디렉토리는 **절대로 읽지 말 것**:

- `frontend/`
- `backend/`
- `design/`
- `docs/`

회의 자료(round/decision/minutes/agenda)만 사용한다. 이 제약이 중요한 이유: 결정문서에 없는 정보가 소스 코드에서 흘러들어가면 정리본이 "회의 결과"가 아니라 "Claude가 추론한 현재 상태"로 변질된다. 회의록은 시점성 문서이므로 회의 시점의 합의만 담아야 한다.

또한 **`decision.md` 본문을 그대로 복붙하지 말 것**. 결정문서는 매트릭스·표·미해결 항목 코드(R-1, U-2 등) 형태로 압축되어 있다. 반드시 팀별 관점으로 재구성하고, 짧은 결정 문장은 풀어쓰기 규칙에 따라 확장한다.

## sumup.md 출력 포맷 (슬림)

`sumup.md`는 **통합 부분만** 담는다. 팀별 섹션은 분리본에 위임한다.

```markdown
# {회의명} — 회의 결과 종합

> 회의: WorKey ...
> 일시: YYYY-MM-DD
> 산출물: round{N}건 + decision.md + minutes.md
> 본 디렉토리 구성:
> - `sumup.md` (이 문서): 통합 마일스톤 표 + 팀 간 의존성 + 한 줄 결론
> - `{team}.md`: 팀별 실행 작업 분리본 (시스템 그림·MUST·SHOULD·백로그)

## 📋 회의 개요

| 항목 | 내용 |
|---|---|
| 안건 | ... |
| 일시 | YYYY-MM-DD |
| 참석 팀 | pm, backend, frontend, design, ux (회의에 따라 동적) |
| 핵심 토픽 | T1 / T2 / T3 / T4 |

## 🗂 분리본 인덱스

| 팀 | 파일 | 주요 작업 (1줄) |
|---|---|---|
| 기획 (PM) | [pm.md](./pm.md) | ... |
| 백엔드 | [backend.md](./backend.md) | ... |
| 프론트엔드 | [frontend.md](./frontend.md) | ... |
| 디자인 | [design.md](./design.md) | ... |
| UX | [ux.md](./ux.md) | ... |

## 📅 통합 액션 아이템 (Action Items)

| # | 액션 | 책임자 | 기한 | 의존성 |
|---|---|---|---|---|
| A1 | ... | backend ↔ pm | 킥오프 +1주 | — |
| A2 | ... | pm | A1 결과 +3일 | A1 |

> `decision.md`의 "다음 액션 아이템" 표를 컬럼 그대로 보존. 기한 정보 압축·삭제 금지.

## 📌 팀 간 의존성 / 미해결 항목

| # | 항목 | 보류 사유 | 결정 시점 / 권고 | 책임 |
|---|---|---|---|---|
| U-1 | ... | ... | ... | frontend |
| U-2 | ... | ... | ... | backend ↔ pm |

## 🎯 한 줄 결론

> "..."
```

`sumup.md`에는 **시스템 그림·JSON 페이로드·팀별 MUST/SHOULD/백로그를 넣지 않는다**. 그건 모두 분리본 책임이다.

## {team}.md 출력 포맷 (팀별 분리본)

각 팀별 파일은 **자기 팀 작업이 자립적으로 보이도록** 구성한다. sumup.md를 보지 않아도 그 팀이 할 일을 끝까지 알 수 있어야 한다.

```markdown
# {팀 이름} — {회의명} 실행 작업

> 출처: meeting-log/{YYYY-MM-DD}-{slug}/sumup.md
> 회의 일시: YYYY-MM-DD
> 본 문서는 sumup의 {팀 이름} 섹션 + 해당 팀 owner 항목만 발췌한 것이다.
> 다른 팀 작업·전체 통합 마일스톤은 [sumup.md](./sumup.md) 참조.

## 시스템 그림 (있으면)
{round 파일의 트리·플로우·ASCII 다이어그램 원문 인용}

## v1.x MUST — 출시 차단 작업
### 1. {기능명}
{풀어쓴 설명 + JSON / 표 / 컴포넌트 수치}
### 2. ...

## v1.x SHOULD — fast-follow
...

## 백로그 (다음 버전)
...

## 의존성 / 미해결 (이 팀 단독)
...

---

## 📅 이 팀 owner 액션 아이템

| # | 액션 | 책임자 | 기한 | 의존성 |
|---|---|---|---|---|
| A3 | ... | design (작성), pm (머지) | 킥오프 +3일 | — |
| A6 | ... | frontend | 킥오프 +1주 | A3 |

> `sumup.md`의 통합 액션 아이템 표에서 책임자 컬럼에 이 팀이 포함된 행만 추출.
> 다른 팀 owner 액션은 sumup.md 통합 표 참조.

## 📌 이 팀 책임 미해결 항목

| # | 항목 | 결정 시점 / 권고 |
|---|---|---|
| U-2 | 인터랙션 매핑 (PM 안 채택) | frontend가 코드에 명시적 채택 |

> `sumup.md`의 팀 간 의존성/미해결 표에서 책임 컬럼에 이 팀이 포함된 행만 추출.
```

각 팀 섹션은 **MUST → SHOULD → 백로그 → 의존성/미해결** 순서를 지킨다. 회의에서 다루지 않은 단계는 해당 단계만 생략한다 (예: SHOULD가 없으면 MUST 다음 바로 백로그).
**단, decision.md가 MUST, SHOULD, 백로그와 같은 포맷으로 작성되어 있지 않다면 decision.md의 포맷과 같이 정리한다.**

### 분리본 추출 규칙

각 분리본은 다음 3개 블록을 자동 조립한다:

1. **팀 섹션 본문** — 시스템 그림 + MUST + SHOULD + 백로그 + 의존성/미해결 (풀어쓰기 규칙 + 보존 규칙 P1~P5 모두 적용)
2. **이 팀 owner 액션 아이템** — `decision.md` 액션 아이템 표에서 `책임자` 컬럼에 팀 이름이 포함된 행만 필터링. 컬럼 구조는 그대로
3. **이 팀 책임 미해결 항목** — `decision.md` 미해결 표에서 `책임` 컬럼에 팀 이름이 포함된 행만 필터링

**필터링 시 주의**: `책임자` 컬럼이 `backend ↔ pm`처럼 두 팀 공동 owner면 **양쪽 팀 모두**의 분리본에 그 행을 넣는다. `frontend + backend`도 마찬가지. 이 경우 한쪽 분리본에서만 보고 다른 쪽이 누락되는 사고가 생긴다.

---

### 풀어쓰기 규칙 (분리본 작성의 핵심)

`decision.md`는 한 줄 결정 형태가 많다. 예:

> `T1-1 BFF GET /me/home-summary v1.2 MUST 도입 (5일 이상 소요 시 frontend client-side aggregation으로 fallback)`

이걸 그대로 옮기면 의미가 전달되지 않는다. 항목 종류에 따라 다음과 같이 풀어 쓴다:

#### A. 엔드포인트
다음 4가지를 모두 포함:
1. **무엇을 하는 API인지** 한 문장 설명
2. **응답 JSON 예시** — round 파일에서 실제 스펙을 찾아 인용. 보통 `backend-round*.md`에 응답 구조가 있다
3. **목적** — 왜 이 엔드포인트가 필요한지 (KPI나 다른 결정과 연결)
4. **fallback / 대체 전략** — 구현 지연 시 어떻게 할지

예시:
```markdown
### `GET /me/home-summary` BFF 엔드포인트 신설
홈 화면 진입 시 3개 API 병렬호출(`/me/onboarding-status`, `/stores/{id}/schedules`,
`/stores/{id}/employees`)을 1회로 통합하는 합산 응답.

```json
{
  "role": "ADMIN",
  "onboardingStatus": "COMPLETE",
  "storeId": 42,
  ...
}
```

- 대용량 데이터(employees·entries 등)는 lazy 별도 호출 유지
- 목적: Cold TTI 1.5s / Warm TTI 0.8s P95 달성
- 5일 이상 소요 시 frontend가 client-side `Promise.all`로 fallback
```

#### B. 컴포넌트 / UI
다음을 포함:
- snapPoints / initialIndex / 위치
- 색상 (헥스 코드 + 토큰명)
- 크기 (dp/pt 단위 명시)
- 인터랙션 동작 (드래그 시퀀스, 햅틱, 토스트 지속 시간 등)

예시:
```markdown
### BottomSheet 4종 (`@gorhom/bottom-sheet` v5)

| 컴포넌트 | snapPoints | initialIndex |
|---|---|---|
| InviteBottomSheet | `['40%']` | 0 |
| LeaveRequestSheet | `['50%','92%']` | 0 |
| DayOffConflictSheet | `['30%','60%','92%']` | 1 (60%) |

backdropOpacity: 0.15(30%) / 0.40(60%) / 0.60(92%). 핸들 색 sky-700, 너비 40dp.
```

#### C. 디자인 토큰
다음을 포함:
- before 값과 after 값
- WCAG 대비 비율 (변경 전/후)
- 변경 이유 (어떤 환경/시나리오 때문인지)

예시:
```markdown
| 토큰 | before | after | WCAG |
|---|---|---|---|
| `--ts-cal-today-bg` | `#0EA5E9` (sky-500) | `#0369A1` (sky-700) | 2.9:1 → 8.75:1 |
```

#### D. 그 외 (정책·KPI·인터랙션 등)

- 결정 코드(R-1, U-2 등)는 출처로 인용하되, 본문은 풀어 쓴다
- 근거가 round 파일에 있으면 적극 활용 (예: "design-round3 §4.2")

---

### 보존 규칙 (Preservation Rules) — 분리본에 적용

풀어쓰기는 한 줄 결정을 확장하기 위한 것이지, **원본 자료의 그림·표·페이로드를 산문으로 압축하라는 뜻이 아니다.** 다음 5가지는 무조건 원본 형태를 보존한다. **이 규칙은 `{team}.md` 분리본에만 적용**한다 (sumup.md는 통합 표만 담으므로 해당 없음).

#### P1. 시스템 그림(Tree / Flow / ASCII)은 원문 그대로 인용

round 파일에 다음과 같은 **전체 그림**이 등장하면 해당 팀 분리본 **상단**에 "## 시스템 그림" 또는 "## 화면 흐름" 서브섹션을 만들어 **원문 형태 그대로 인용**한다. 절대 분해해서 풀어쓰지 말 것.

대상:
- 트리 구조도 (예: `RootNavigator ├── AuthStack ├── MainStack ...`)
- 화면 흐름도 / 플로우차트
- ASCII 화면 레이아웃
- 모듈/패키지 구조 다이어그램
- 네트워크/데이터 흐름도

이유: 이런 그림은 신규 합류자가 시스템 전체를 한눈에 조망하는 단일 진입점이다. 산문으로 풀면 즉시 가치를 잃는다.

#### P2. JSON 페이로드는 단계 무관하게 보존

round 파일에 **엔드포인트 + JSON 예시**가 함께 등장하는 항목은 **단계(MUST / SHOULD / 백로그)와 무관하게** JSON 1건 이상을 분리본에 포함한다.

- MUST는 풀어쓰기 규칙 A 4가지(설명·JSON·목적·fallback) 모두 포함
- SHOULD/백로그도 round 파일에 JSON이 있으면 **최소 1건 인용** (한 줄 요약 + JSON 블록)
- JSON이 round 파일에 없는 엔드포인트만 한 줄 요약 허용

이유: SHOULD/백로그라도 회의에서 합의된 contract이다. 회의록에 박혀 있어야 PR 시점에 round 파일을 다시 뒤지지 않는다.

#### P3. 토큰/수치 표는 표 형태로 보존 (코드로 대체 금지)

round 파일에 **3행 이상의 토큰/수치 매핑 표**가 있으면 **표 그대로** 옮긴다. 코드 스니펫(`tokens.ts` 등)이 함께 있어도 **표를 코드로 대체하지 말 것**.

- 표는 디자이너 검수용 (시각적 비교)
- 코드는 개발자 구현용 (복붙)
- 둘 다 필요하면 둘 다 적는다

대상 예: 다크 모드 토큰 매핑 N종, FAB 일반/매장 모드 비교 표, BottomSheet 컴포넌트별 snap·backdropOpacity·핸들 색 표.

#### P4. 통합 액션 아이템 표 강제 보존 (sumup.md)

`decision.md`에 "다음 액션 아이템", "Action Items", "후속 작업"처럼 **책임자·기한이 컬럼으로 정리된 표**가 있으면, `sumup.md`에 "## 📅 통합 액션 아이템" 섹션으로 **표 그대로** 보존한다.

- 위치: "팀 간 의존성 / 미해결 항목" 표 **위**
- 컬럼: 최소 `# / 액션 / 책임자 / 기한` 4개 유지. `의존성` 컬럼이 원본에 있으면 함께 유지
- 기한 정보는 압축·삭제 금지 ("킥오프 +1주", "+3일", "v1.2 출시 1주 전" 등 그대로)

분리본에는 **이 팀 owner 행만 추출한 부분 표**가 들어간다 (위 분리본 추출 규칙 참조).

이유: PM이 마일스톤을 팔로업할 때 시간축으로 정렬된 단일 표가 필요하다. 분리본에만 흩어 두면 추적 불가능해진다.

#### P5. 풀어쓰기 ≠ 압축

풀어쓰기 규칙은 한 줄 결정(`T1-1 BFF GET ...`)을 4가지 항목으로 **확장**하라는 뜻이다. 반대로:

- round 파일의 N행 표를 1-2줄 산문으로 줄이는 것은 **압축**이며 금지
- round 파일의 ASCII 트리를 "5레이어 구조"라고 한 줄로 적는 것은 **압축**이며 금지
- round 파일의 JSON 블록을 "응답 필드는 role, storeId 등"이라고 적는 것은 **압축**이며 금지

원본보다 정보량이 줄어들면 풀어쓰기가 아니다.

### 날짜(YYYY-MM-DD) 추출 우선순위

1. `decision.md` 본문의 `일시:` 필드 (예: `| **일시** | 2026-04-29 |` 또는 `일시: 2026-04-29`)
2. `minutes.md` 본문의 `일시:` 필드
3. 둘 다 없으면 사용자에게 직접 질문 ("회의 일시를 YYYY-MM-DD로 알려주세요")

### 슬러그 규칙

- 디렉토리명에서 `-meeting` 접미사가 있으면 **제거**한 뒤 사용
  - `frontend-kickoff-meeting/` → 디렉토리명: `2026-04-29-frontend-kickoff/`
- 디렉토리명이 한글을 포함하면 사용자에게 영문 슬러그를 1회 질문
  - `프론트엔드-킥오프/` → "영문 슬러그를 알려주세요" 질문 → 사용자 답변 사용
- 그 외 영문 kebab-case 디렉토리명은 그대로 사용
- **`-summary` 접미사는 더 이상 붙이지 않는다** (분리본 구조에서는 의미가 없다)

### 저장 흐름 (반드시 이 순서)

1. **터미널에 sumup.md 본문을 먼저 출력**한다 — 사용자가 통합 결정 사항을 검토할 수 있어야 한다. 분리본은 양이 많으므로 본문 출력 대신 **파일명·헤더·MUST 항목 수만 요약** 출력 (예: "backend.md: 시스템 그림 1 + MUST 7 + SHOULD 8 + 백로그 5")
2. Obsidian 동일 디렉토리(`{YYYY-MM-DD}-{slug}/`)가 이미 있는지 `mcp__obsidian__list_directory`로 확인
3. 디렉토리가 이미 있으면 사용자에게 **"기존 디렉토리에 파일 N개가 있습니다. 모두 덮어쓸까요?"** 확인. **무단 overwrite 금지**. 디렉토리 안의 어떤 파일이라도 덮어쓰는 것은 사용자 확인 필수
4. 사용자 확인을 받은 뒤 다음 순서로 작성:
   a. `sumup.md` 작성
   b. 회의 참여 팀 각각 `{team}.md` 작성 (감지된 prefix 순서대로)
5. 저장 완료 후 모든 파일 경로를 사용자에게 보고 (디렉토리 + 파일 목록)

### 사용 도구

- `mcp__obsidian__list_directory` — `10. Projects/WorKey/meeting-log/` 존재 및 기존 디렉토리 확인

## 작업 순서

1. 사용자 메시지에서 `{안건}` 추출 (예: "frontend-kickoff-meeting 회의 정리해줘" → `frontend-kickoff-meeting`)
2. `meeting/{안건}/` 디렉토리 존재 확인. 없으면 사용자에게 알리고 중단
3. 회의 자료 파일 수집:
   - `agenda.md`, `decision.md`, `minutes.md`
   - `evidence/*-round*.md`, `rebuttals/*-round*.md` 모두 (글롭)
4. round 파일 prefix로 참여 팀 감지 (5팀 중 일부만 있을 수 있음)
5. `decision.md`의 `일시:` 필드 추출 → 없으면 `minutes.md` → 없으면 사용자 질문
6. **보존 자산 인덱싱** (분리본 작성 전 필수 단계):
   - round 파일에서 **시스템 그림**(트리·플로우·ASCII 다이어그램) 위치 표시
   - round 파일에서 **JSON 페이로드** 블록 모두 수집 (엔드포인트별)
   - round 파일에서 **3행 이상의 표** 모두 수집 (토큰·수치·컴포넌트)
   - `decision.md`에서 **액션 아이템 표** 위치 표시
   - `decision.md`에서 **미해결 항목 표** 위치 표시
7. **sumup.md 작성** — 회의 개요 + 분리본 인덱스 + 통합 액션 아이템 표 (P4 그대로 보존) + 팀 간 의존성/미해결 표 + 한 줄 결론. **팀별 풀어쓰기·시스템 그림·JSON은 절대 넣지 않는다**
8. **분리본 작성** — 회의 참여 팀 각각에 대해:
   - 팀 섹션 본문 (시스템 그림 → MUST → SHOULD → 백로그 → 의존성) — 풀어쓰기 규칙 + 보존 규칙 P1~P3, P5 적용
   - 이 팀 owner 액션 아이템 부분 표 (필터링)
   - 이 팀 책임 미해결 항목 부분 표 (필터링)
9. **자가 점검**:
   - [ ] sumup.md에 팀별 MUST/SHOULD가 들어가지 않았나? (들어갔으면 분리본과 redundant)
   - [ ] 분리본에 시스템 그림·JSON·표가 원문 그대로 들어가 있나?
   - [ ] 공동 owner 액션 아이템(`A ↔ B`)이 양쪽 팀 분리본 모두에 들어갔나?
   - [ ] 회의 참여 팀 수와 분리본 파일 수가 일치하나?
   - [ ] sumup.md `🗂 분리본 인덱스`의 파일 링크가 실제 작성될 파일과 일치하나?
10. **터미널에 sumup.md 본문 + 분리본 파일 목록·요약 출력** (사용자 검토용)
11. Obsidian 저장 디렉토리 결정 (`{YYYY-MM-DD}-{slug}/`)
12. 동명 디렉토리 존재 시 **덮어쓰기 여부 사용자 확인** (디렉토리 단위, 안의 파일 모두 영향)

14. 사용자에게 디렉토리 + 모든 파일 경로 보고

## 주의사항 (자주 빠지는 함정)

- **소스 코드 금지 재확인**: round 파일에 코드 스니펫이 있어도 그건 회의에서 합의된 스펙이다. 그러나 `frontend/`·`backend/`·`design/`·`docs/`를 직접 읽지는 말 것
- **결정문서 한 줄 = 결과물 한 줄 아님**: `decision.md`의 한 줄짜리 결정은 풀어쓰기 규칙으로 확장한다. 같은 길이로 옮기면 이 스킬을 쓰는 의미가 없다
- **팀 5개 고정 아님**: round 파일 prefix만 보고 동적으로. 회의에 디자인이 안 왔으면 `design.md` 분리본 자체를 만들지 말 것
- **저장 전 미리보기 필수**: 사용자 확인 없이 Obsidian에 쓰지 말 것. 분리본은 분량이 크고 디렉토리 단위로 한 번에 덮어쓰면 기존 회의록이 통째로 날아갈 수 있다
- **풀어쓰기에 매몰돼 그림·표·JSON을 잃지 말 것 (가장 자주 빠지는 함정)**: 분리본에서 round 파일의 Navigation 트리·다크 모드 토큰 매핑 표·FAB 수치 표·JSON 응답 예시 같은 **전체 그림과 세부 스펙**을 산문으로 압축하면 안 된다. 보존 규칙 P1~P5를 매번 확인하라
- **sumup.md를 다시 sumup-of-sumup으로 만들지 말 것**: sumup.md는 통합 마일스톤 표(액션·미해결)만 담는다. "팀별 핵심 작업 요약" 같은 섹션을 추가하면 분리본과 정보가 중복돼 다시 컨텍스트가 폭발한다. 팀별 정보는 무조건 분리본에 위임
- **공동 owner 행 누락**: 액션 아이템 책임자가 `backend ↔ pm`이면 backend.md와 pm.md **양쪽** 분리본에 그 행을 넣어야 한다. 한쪽만 추출하면 다른 팀이 자기 작업을 모른 채 진행하는 사고가 생긴다
- **분리본을 작업 시 컨텍스트로 활용**: 사용자가 "각 팀 문서 반영"을 요청할 때는 해당 팀 분리본만 클로드에게 던지면 된다. sumup.md는 PM·team-lead용으로만 쓰고 작업 반영 시에는 굳이 같이 던지지 않는다 (스킬 사용자에게 안내할 가치)
