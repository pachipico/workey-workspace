작성자: ux (최기준 점장 페르소나)
회의: frontend-kickoff-meeting
참조: `docs/WorKey - 기획서.md` (§3.1, §3.3, §4.1, §10), `docs/WorKey - 디자인.md` (§휴무 수집 뷰, §화면 목록)

> 한마디로 — "지금 이대로 매장 카운터에 서서 한 손으로 일정 짠다고? 두 번에 한 번은 카톡으로 도망친다."

---

## 0. 페르소나 — 최기준 점장 (현실 모드)

- 38세, 분식+카페 복합 매장 점장. 직원 8명(앱 연동 4 / 미연동 4 — 현실 혼합).
- 오른손은 카드 단말기·포스, 왼손에 폰. 영업 중에 일정 작업을 끼워 넣는다.
- 노안 시작, 매장 형광등 직사광에 화면 잘 안 보임.
- "두 탭 이상 헤매면 그냥 카톡 사진으로 돌아갈 거야."
- 푸시는 이미 배달앱·POS·은행으로 하루 30개 넘게 받는다. WorKey 알림 1개 늘어나는 거 의미 있는 건지 의심부터 한다.

## 1. 한 손으로 일정 등록을 완수할 수 있는가? — 시나리오별 분석

### 시나리오 A: 다음 달 스케줄 자동 생성 (관리자 핵심 플로우, §3.3)

플로우 단계 단위로 한 손 가능성 평가.

| # | 단계 | 한 손 가능 여부 | 코멘트 (점장 시점) |
|---|------|------------------|-------------------|
| 1 | 대시보드 → "스케줄 생성" 진입 | ⭕ 가능 | 진입점 1탭은 OK. 단, 진입 버튼이 화면 상단이면 망함 — Bottom 영역에 있어야 함. |
| 2 | 조건 입력 (대상 월 / 최소 인원 / 잉여 휴무 옵션 / StoreHoliday) | ⚠️ 위험 | 입력 필드가 4개 이상 위→아래로 깔리면 키보드 띄울 때 위 필드는 가려진다. 한 손으론 스크롤하며 키 누르기가 끔찍. |
| 3 | "스케줄 생성" → 휴무 수집 뷰 진입 | ⭕ 가능 | 1탭 전환은 OK. |
| 4 | 휴무 수집 뷰 충돌 셀 클릭 → 바텀시트 → 칩 드래그 | ❌ 거의 불가 | **이 단계가 한 손 시나리오의 사망 지점.** 칩을 잡고 손가락 끝까지 끌어 다른 셀에 떨어뜨리는 동작은 엄지 단독으로 정확도가 너무 떨어짐. 디자인 §휴무 수집 뷰 기준 칩 크기·셀 크기·손가락 가림 모두 고려 안 된 상태. |
| 5 | `+ 휴무 수동 추가` → 직원 선택 → 다중 날짜 선택 | ⚠️ 위험 | 다중 선택은 한 손에 가능하지만 직원 모달이 풀스크린이면 닫기 X 버튼 위치(상단 우측)가 한 손에 안 닿는다. |
| 6 | 충돌 0 확인 → "생성 확정" | ⭕ 가능 | 단, "생성 확정"이 헤더에 있으면 가짜 한 손 — 상단 도달 불가. |
| 7 | 자동 생성 → 수동 조정(드래그&드롭) | ❌ 어려움 | 4번과 동일한 드래그 정확도 문제. |
| 8 | 출력(PNG) → 카톡 공유 | ⭕ 가능 | OS 공유 시트 호출 1탭이면 끝. |

**결론**: 4·7번이 깨지면 점장은 "그냥 종이 뽑아서 적는다"로 회귀. 드래그를 한 손 친화 동작(롱프레스 → 메뉴 → "다른 날로 옮기기" 선택)으로 **fallback** 제공이 필수.

### 시나리오 B: 직원 1명 신규 등록 + 초대 링크 발송 (§4.1)

- 직원 등록 모달의 입력 칸: 이름, 이모지(프로필), 월 휴무 개수, 선호 패턴 — 4개. 한 손 가능하지만 이모지 선택 UI가 그리드면 엄지로 모서리 이모지 못 누름.
- 초대 링크: "직원 상세 → 초대 바텀시트 → 카카오톡 전송" — 3탭. 카카오 공유 시트가 OS 레벨에서 뜨므로 그 다음은 OS 책임. 여기까진 OK.

**결론**: 직원 등록 자체는 한 손 가능. 단, 이모지 그리드는 위쪽 행을 한 손이 못 누르는 문제 있음 → 검색·횡스크롤 권장.

### 시나리오 C: 미연동 직원 휴무 5건 일괄 입력 (§4.1 휴무 수집 뷰)

- `+ 휴무 수동 추가` → 직원 선택 → 달력에서 5개 날짜 다중 선택 → 확인.
- 한 달 달력은 보통 6주 그리드. 1주차 / 6주차 셀은 화면 상·하단 끝 → 엄지로 못 닿음.
- **점장은 이 작업을 "비번 5명분 카톡 정리해서 한꺼번에 입력"하는 시나리오로 사용한다.** 한 손이 막히면 이 시나리오 전체가 막힘.

## 2. "내가 점장이라면 이 단계에서 짜증날 것 같다" — 페인 포인트 (≥5)

> 톤은 점장 본인 1인칭. 디자인·기획·프론트 모두 들으라고 쓴 것임.

### PP-1. **드래그&드롭이 손가락에 가려서 어디 떨어지는지 모르겠다.**
> "휴무 수집 뷰에서 김민준 칩 잡고 17일에서 23일로 옮기려는데, 손가락이 칩이랑 셀 다 가려서 어디 놨는지 보이지도 않아. 떨어뜨리고 나서야 21일에 잘못 떨어진 걸 알았다. 그럼 또 잡아서 옮긴다? 영업 중에 이거 두 번 하면 폰 던지고 싶다."
- 위치: 디자인 §휴무 수집 뷰, 드래그&드롭 동작 규칙
- 요구: 드래그 중 칩의 미러 표시(손가락 위로 floating preview), 드롭 직전 셀 강조 + 햅틱, 잘못 놨을 때 1회 Undo 토스트(5초)

### PP-2. **충돌 빨간 표시가 "왜" 빨간지 한 화면에서 안 읽힌다.**
> "17일이 빨개. 빨간 게 뭔지 안다. 근데 누구 신청 때문에 빨갛게 됐는지를 보려면 그 셀을 또 눌러야 해. 그리고 바텀시트가 올라오면 달력이 반쯤 가려져서 옆 날짜 상황이랑 비교가 안 돼. 결국 바텀시트 닫고 다시 누르고를 반복한다."
- 요구: 충돌 셀 long-press → 인라인 미니 팝오버(달력 위 60% 높이만)로 신청자 미리보기, 바텀시트는 풀하이트 대신 50% 높이 기본

### PP-3. **무료·스탠다드 티어인데 "+ 수동 추가"가 유일한 입력인지 어디서도 확실하게 안 알려준다.**
> "처음 써보면 '내 직원들이 앱 안 깔았는데 이거 어떻게 쓰지?' 하다가 + 버튼을 우연히 발견한다. 발견 못 하면 그냥 종이로 돌아간다."
- 위치: §3.3·디자인 §휴무 수집 뷰
- 요구: 티어가 팀이 아니거나 연동 직원 0명일 때 휴무 수집 뷰 진입 직후 1회 코치마크 ("이 매장은 직접 입력해야 합니다 — 여기를 누르세요" + + 버튼 펄스)

### PP-4. **공유한 PNG가 카톡에서 잘려서 어차피 줌인해야 한다.**
> "8명 × 30일 표를 카톡에 올리면 글자가 깨알. 직원이 '내 이름이 안 보여요' 카톡한다. 그럼 또 줌인 캡처해서 다시 올린다. 이럴 거면 그냥 ICS·xlsx 줘. 근데 그건 스탠다드부터…"
- 위치: §6 출력 형식
- 요구: PNG 출력 시 "1주씩 4장 + 전체 1장" 멀티 이미지 옵션. 무료 티어에도 이 옵션은 줘야 한다.

### PP-5. **푸시 알림이 너무 잦으면 다 끈다 — 그러면 앱 의미가 없어진다.**
> "직원 8명 매장에서 휴무 신청, 신청 변경, 근무 교환 요청, 근무 교환 수락, 관리자 당일 알림(매일 오전 8시), 직원 쉬는 전날 알림(저녁 8시) — 합쳐서 하루 5~7개 푸시. 일주일 만에 'WorKey 알림 끔' 누른다."
- 위치: §4.1 관리자 당일 알림 / §4.2 쉬기 전날 알림 / §4.2 휴무·교환 푸시
- 요구: 알림 묶음(daily digest) 토글 — "오늘 변경사항 1건 모아 한 번에", 카테고리별 on/off, 영업시간(예: 11–14, 17–20) 자동 무음 옵션. **온보딩 단계에서 푸시 권한 즉시 요청 금지** — 첫 휴무 신청 도착 직전에 컨텍스트 있게 요청.

### PP-6. **딥링크로 들어왔는데 카카오 OAuth부터 띄우면, 점장은 직원 핑계 듣다 폭발.**
> "직원이 '링크 눌렀더니 갑자기 카카오 로그인 뜨고 다음에 또 뭐 누르래요' 하면 내가 옆에서 같이 봐줘야 한다. 영업 중에. 디지털 친숙도 낮은 직원 1명만 있어도 매장 운영 멈춘다."
- 위치: §3.2 케이스 3, §10.1 딥링크
- 요구: 초대 링크 진입 시 환영 화면 → "다음" 1번 누르면 카카오 OAuth, 이전에 매장명·점장 이름·"이 가입은 1분 안 걸려요" 메시지 노출. 카카오 OAuth는 미리 설명 후 트리거.

### PP-7. **조건 설정에서 "잉여 휴무 옵션 — 균등 분산 / 연속 부여"라는 단어가 무슨 말인지 점장은 모른다.**
> "균등? 연속? 그냥 '몰아서 쉬게 / 골고루 쉬게' 라고 적어. 우리 매장 직원이 몰아서 3일 쉬는 게 좋은지 흩어서 쉬는 게 좋은지를 묻는 거잖아."
- 위치: §3.3 / §4.1 조건 설정
- 요구: 라벨을 점장 언어로 바꾸고, 각 옵션 아래 1줄짜리 결과 미리보기 ("이번 달 김민준: 휴무 6일을 17,18,19,25,26,27 두 덩어리로" / "1·5·9·13·19·25일 분산").

## 3. 알림 피로도 (Notification Fatigue)

- **현재 알림 카탈로그(추정)**: 휴무 재배치 통지(직원), 근무 교환 요청·수락·승인(양측), 쉬는 전날(직원), 관리자 당일 8시(관리자), 향후 직원→관리자 메시지(검토 중).
- **위험**:
  1. 같은 사건이 여러 푸시로 쪼개진다(예: 휴무 1건 재배치 → 직원 1명 1푸시 = OK, but 5명 동시 재배치 시 5푸시 연속 발사).
  2. 매장 영업 시간(점심 피크 11–14)에 알림이 와도 점장은 못 본다 → 시간 지나서 한꺼번에 보면 "어떤 게 최신인지" 모름.
  3. 직원이 알림 끄면 "휴무 변경 통지 못 받았다"는 분쟁 발생 → 점장이 카톡으로 또 보낸다 → WorKey 의미 없음.
- **권고**:
  - **알림 그룹화**: Android는 channel + group, iOS는 thread-id 활용해 매장별·카테고리별 묶기.
  - **Daily digest 모드**: "휴무 5건 재배치, 교환 1건 승인" 한 줄 요약 1회.
  - **영업시간 무음**: 매장 운영 시간 입력받아 그 시간엔 silent push로 변경(데이터만 도착, 알림 음 X).
  - **권한 요청 타이밍**: 앱 첫 실행 즉시 X. 첫 휴무 신청·교환 요청 직전에 컨텍스트 있게 요청.
  - **카카오톡 대비 가치 명시**: 푸시 받기로 하면 "카톡 안 봐도 됨"이라는 메시지를 첫 권한 요청 화면에 명시.

## 4. 오인식 위험 (Mis-tap / Mis-action)

| 위험 | 시나리오 | 대응 |
|---|---|---|
| **드래그 오드롭** | 17일 칩을 18일에 떨어뜨리려 했는데 25일에 떨어짐 — 손가락에 가려서 셀이 안 보임 | 드롭 직후 토스트 + 5초 Undo, 햅틱, drop preview floating |
| **삭제된 직원 안내 오인식** | 직원 입장에서 "현재 연결된 매장이 없어요"라는 문구가 "내 잘못으로 매장이 사라졌다"로 읽힘 | 디자인 §화면 목록의 안내 카피를 점장 행위 명시("[매장명] 관리자가 회원님을 직원 명단에서 내렸어요. 관리자에게 문의하세요")로 변경 |
| **매장 변경 확인 오탭** | "[새 매장명]으로 바꿀까요?" → 무심코 확인 → 기존 매장 비활성화 → 직원 충격 | 확인 버튼은 1차 컨펌 + 2차 "[기존 매장명]에서 빠집니다. 진행할까요?" 더블 컨펌. Destructive 색상 톤. |
| **"생성 확정" vs "수동 조정 저장"** | 두 화면이 비슷하게 생기면 점장이 "확정"을 "저장"으로 착각해 자동 생성을 트리거 | 확정·저장 버튼은 위치(좌/우)·색·아이콘 모두 분리, 확정 시 1회 컨펌 시트 |
| **"생성 월 2회" 무료 한도 초과 직전 무경고** | 점장이 두 번째 생성 → 생성 후에 "이번 달 한도 끝" 알림 → 다시 생성 못 함 → 분노 | 생성 버튼 누르기 전 한도 잔여 표시 ("이번 달 1회 남음"), 마지막 생성 직전 명시적 컨펌 |
| **푸시 권한 즉시 요청** | 온보딩 1초만에 권한 다이얼로그 → 점장 거절 → 이후 권한 회복 어려움 | §3 권고대로 컨텍스트 시점 요청, 거절 후에도 in-app 배너로 재유도 |

## 5. 인지 부하 평가 — 매장 환경 변수

### 환경 변수
- **소음**: 분식·카페 환경 70~80 dB. 음성 피드백/벨소리 알림 무의미.
- **조명**: 형광등 + 외부 직사광 → 화면 명도 대비 7:1 미만 글자 안 읽힘.
- **시간 압박**: 손님 응대 사이 30~90초 자투리에 작업.
- **물리 간섭**: 손이 젖거나 기름 묻은 채로 터치 → 정전식 인식 불안정.

### 인지 부하 지점

| 지점 | 부하 원인 | 완화 |
|---|---|---|
| 휴무 수집 뷰 충돌 셀 | 빨간 셀 + 칩 + 드래그 + 바텀시트 동시 노출 → 동시 처리 정보 4개 | 1단계: 충돌 셀 long-press → 미니 팝오버 (1정보) → 사용자가 "재배치" 액션 명시 후 칩 모드 |
| 조건 입력 | 4개 필드 + 라벨이 점장 언어 아님 + 미리보기 없음 | 진행형 위저드(1필드/1화면)와 결과 미리보기, "권장값" 자동 채움 |
| 알림 도착 | 무엇이 새로 변했는지 즉시 인지 어려움 | 알림 그룹 + in-app "변경사항 N건 보기" 배너 |
| 색약·노안 | sky-500 #0EA5E9 단독 의존 시 ON/OFF 구분 어려움 | 색 + 아이콘 + 텍스트 라벨 3중 인코딩 (디자인 §Primary 색상) |
| 시각 명도 | 매장 직사광에서 sky-50 #F0F9FF 면이 흰색과 거의 동일 | "high-contrast" 보조 모드(설정 토글) — 매장 환경용 |

### 권고 — "매장 모드(Store Mode) 토글"
- 토글 ON 시: 폰트 크기 +1단계, 컨트롤 hit area +20%, 전 면 contrast 강화, 알림 silent push, "한 손 모드" UI 우측 정렬.
- 점장 온보딩에서 1회 안내. 디자인 시스템 토큰으로 분기 가능해야 함(디자인 측 검토 필요).

## 6. 4대 핵심 토픽에 대한 깐깐한 사용자 시각

### T1. 모바일 최적화 API (초기 로딩 / 응답 구조)
- **점장 요구**: "앱 켜고 0.8초 안에 이번 달 스케줄 보여라. 그 이상 걸리면 앱 닫고 카톡 본다."
- **현실 페이로드 우려**: 대시보드 진입 시 (1) 매장 정보 (2) 이번 달 스케줄 (3) 직원 명단 (4) 휴무 신청 카운트 (5) 위반 여부 — 5건을 별도 콜이면 모바일 LTE 환경에서 합산 1.5초+ 가능.
- **요구**:
  - **Aggregated bootstrap endpoint** (`/me/dashboard?storeId=...`) — 위 5건을 1콜에 합쳐 응답.
  - **stale-while-revalidate** — 마지막 캐시 결과 즉시 노출, 백그라운드 재검증.
  - **연간 통계 (`GET /stores/{storeId}/schedules/annual`, §4.1)** 같이 잘 안 변하는 데이터는 ETag/304 적극 활용.
  - 휴무 수집 뷰의 충돌 재검증은 **클라이언트 단**에서 즉시 계산하고, 서버는 최종 확정 시점에만 검증 (드래그 도중 매번 API 호출 금지).
- **백엔드에 던질 질문(Round 2 후보)**: bootstrap aggregation 가능 여부 / 휴무 수집 뷰 충돌 검증 클라이언트화 시 sealed 검증을 어디까지 서버 권한으로 할지.

### T2. 네이티브 연동 (푸시 / 로컬 스토리지)
- **점장 요구**: "오프라인에서도 이번 달 스케줄은 봐야 한다. 매장 와이파이 약하니까."
- **요구**:
  - **로컬 캐시**: 이번 달·다음 달 스케줄 + 직원 명단 + 매장 설정을 SQLite 또는 MMKV(빠른 KV)에 영구 저장. 앱 cold start 시 캐시 즉시 렌더, 네트워크 응답으로 patch.
  - **푸시 인프라**: §3 알림 피로도 권고 적용. APNs/FCM thread/group + silent push 기반 in-app 배지 갱신 + 영업시간 silent 처리.
  - **딥링크**: §10.1 Universal Links / App Links — 미설치 기기에서도 deferred deep link로 토큰 소비. 첫 실행 시 자동 매장 연결 보장.
  - **백그라운드 갱신**: 직원 앱은 새벽 04:00~06:00 사이 1회 백그라운드 fetch로 다음 날 알림 정확도 확보(저녁 8시 푸시 직전 데이터 stale 방지).
- **frontend에 던질 질문(Round 2 후보)**: 로컬 DB로 SQLite vs MMKV vs WatermelonDB 어떤 걸 선택할 것인지, RN 환경에서 영업시간 silent push 구현 책임이 OS인지 서버인지.

### T3. 디자인 이식 (BottomSheet 등)
- **점장 요구**: "BottomSheet가 화면을 다 덮어버리면 그 뒤 달력이 안 보여서 비교가 안 된다."
- **요구**:
  - **BottomSheet 높이 단계**: 30% / 60% / 풀하이트 3단. 충돌 정보는 60% 단에서 시작 — 달력 상단 절반 보이게.
  - **Pull handle 가시성**: 상단 핸들 너비 ≥ 40dp, 색 대비 4.5:1.
  - **드래그&드롭 in BottomSheet**: 칩이 바텀시트 안에 있는데 드롭 대상은 바깥 달력 — 시트 자동 50% 축소 + 드롭 가능 셀 외곽 파란 테두리. 디자인 §휴무 수집 뷰 규칙 보강 필요.
  - **세이프 영역 / 노치 / 홈 인디케이터**: iOS Bottom Tab + 홈 인디케이터 22pt 침범 금지.
  - **다크모드**: 매장 형광등 환경에서 라이트 모드가 더 잘 보이는 경우가 많음 — 시스템 다크 따라가지 말고 앱 내 설정 분리.
- **design에 던질 질문(Round 2 후보)**: BottomSheet 3단 높이 RN 라이브러리 표준화 / 드래그 중 시트 자동 축소 인터랙션 정의 / 매장 모드(고대비) 토큰 스펙.

### T4. 사용자 환경 (한 손 시나리오)
- **점장 요구**: §1 시나리오 A의 4·7번이 한 손에 가능해야 한다.
- **요구**:
  - **Bottom Tab 네비게이션 우선**(상단 탭 비추) — 엄지 도달 영역(thumb zone) 안.
  - **CTA 위치 규칙**: 핵심 액션(생성 확정·저장·다음)은 bottom-fixed full-width, hit area ≥ 56dp.
  - **드래그 fallback**: 칩 long-press → 메뉴 → "이 직원의 휴무를 다른 날로 옮기기" → 달력 셀 1탭 — **드래그 없이도 동일 결과**.
  - **상단 X / 닫기 → 좌상단 + 모달 dismiss 제스처 + 시스템 백 버튼** 3중 제공.
  - **한 손 모드 토글**: 화면 우측 정렬, 키패드·드롭다운도 우측 기준.
- **frontend에 던질 질문(Round 2 후보)**: Bottom Tab vs Top Tab 결정 기준, 드래그 라이브러리(react-native-reanimated + gesture-handler) 한 손 fallback 구현 비용, RN 햅틱 일관성(iOS/Android 차이).
