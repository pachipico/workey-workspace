## 0. 회의 목적 재확인

`docs/WorKey - 기획서.md` §8 로드맵 v1.2 Backlog 기준, 백엔드는 v1.0/v1.1까지 SHIPPED 상태이며 **프론트엔드는 아직 미착수**다. 이번 킥오프는 프론트 첫 스프린트의 스코프·우선순위를 확정하기 위한 자리이며, PM 입장에서는 다음 세 가지가 핵심이다.

1. v1.2 미착수 항목(휴무 수집 뷰 UI / 직원 모바일 앱 UI / 푸시 인프라) 중 **무엇을 가장 먼저 출시할 것인가**

## 2. 가장 큰 리스크 — 휴무 수집 뷰 (§3.3)

PM이 이번 스프린트에서 **가장 강하게 정렬을 요청하는 화면**. 이유:

1. 기획서 §3.3에서 "직원 앱 연동 여부와 무관하게 하나의 화면으로 수렴"이라고 명시. 즉 무료/스탠다드/팀 모든 티어가 같은 화면을 본다 — **티어 분기 UI 없음, 연동/미연동 칩만 다른 시각화**.
2. 충돌 검증은 **드래그&드롭 후 실시간 재검증** 요구 — FE 단독 클라이언트 검증인지, BE 호출인지 합의 필요.
3. 모든 충돌 해소 시에만 "생성 확정" 활성화 — 또는 "경고 후 강행 옵션" 중 어떤 정책을 채택할지 §3.3 본문이 양자 택일로 남겨두었음. **PM은 1차 출시는 "강행 옵션 없음(블로킹)"** 권장 — UX 단순화·오류 신고 감소 목적.
4. `confirmedDayOffs` 페이로드 형태가 §4.1에서 "값 있으면 우선, 없으면 APPROVED 자동 반영"으로 명시 — 프론트는 어떤 경우에도 명시적으로 전송하는 정책을 권장(ambiguity 차단).

## 4. 온보딩 5가지 케이스(§3.2) — 프론트 분기 매트릭스

기획서 §3.2 표는 **프론트 라우팅 분기의 원본**이다. 5가지 모두 첫 진입에서 분기되어야 하며, 누락되면 v1.2 출시 불가.

- 케이스 1·2는 v1.2 P0(관리자/직원 신규 진입)
- 케이스 3·4·5는 직원 앱 P2 트랙으로 분리해도 무방하나, **케이스 4(소속 종료 안내)**는 직원 앱 출시 동시에 반드시 포함 (소송 리스크/CS 부담)

## 5. 알고리즘 응답 UX 정책 (§5)

기획서 §5.1·§5.2에 의하면 스케줄 생성 응답은 다음 3가지로 갈라진다.

| 응답 유형 | FE UX 권장 |
|---|---|
| 사전 검증 실패 → 예외, DB 저장 없음 | **에러 모달**, 직원 수/dayOff/minStaff 중 어느 조건이 깨졌는지 명시. 휴무 수집 뷰로 되돌리기. |
| 생성 성공 + violations 비어있음 | 정상 캘린더 표시 |
| 생성 성공 + violations 존재 (`violated=true`) | 캘린더 위 **노란 경고 배너** + 수동 조정 유도. 6종 violation 코드별 한국어 카피 필요(P0). |

**PM 요청**: 이번 라운드에서 BE 측에 violations 코드별 메시지 사전 정의를 부탁하고, FE는 메시지 카탈로그를 한 곳에서 관리.

## 6. PM이 Round 2에서 다른 팀에게 던질 핵심 질문 예고

(Round 2에서 SendMessage로 정식 전달 예정)

- **frontend**: 휴무 수집 뷰의 드래그&드롭 충돌 재검증을 **클라이언트 단독 계산**으로 처리할 의사가 있는가? 아니면 매 드롭마다 BE를 호출할 계획인가? 모바일 웹 성능 영향이 우려됨.

## 7. PM 결론

- **v1.2 1차 출시 스코프 = §1의 P0 + P1.** P2(직원 앱) 동시 출시는 일정상 무리이며, 기획서 §7도 팀 티어로 분리되어 있어 비즈니스 기조와 정합.
- **휴무 수집 뷰가 단일 최대 리스크** — Round 2에서 FE/BE에게 충돌 검증 책임 분담을 반드시 합의받아야 함.
- 프론트는 "디지털 친숙도 낮은 점장"(§9 리스크 2번)을 1순위 페르소나로 두고, 의심스러우면 단순화 쪽으로 결정.
