## 0. 요약 (Executive Summary)

- **현재 상태**: 백엔드 v1.1 SHIPPED (2026-04-28). `§4.1 + §4.2` 핵심 기능 20개는 모두 백엔드 구현 완료. **남은 가치는 100% 프론트엔드(RN)에 달려 있다.**
- **MVP의 본질**: v1.2의 RN 앱은 "팀 티어(월 19,900원)" 정당화 도구. 직원 앱이 작동하지 않으면 가장 비싼 티어가 비어있게 된다.
- **킥오프의 4대 토픽 결론(PM 입장)**:
  1. **모바일 최적화 API** → 백엔드는 끝났으므로, 추가 개선보다 **`/me/onboarding-status` 등 통합 엔드포인트 + 페이로드 슬림화**로 첫 화면 TTI를 잡는다.
  2. **RN 네이티브 연동** → **FCM 푸시 + AsyncStorage(JWT/오프라인 큐)** 두 축은 MVP 필수. 딥링크는 도메인 발급 일정상 Fast-follow.
  3. **디자인 이식** → BottomSheet 4개 화면(휴무 신청·근무 교환·직원 초대·충돌 해소)이 sky-500 토큰 체계와 1:1 일치해야 한다.
  4. **한 손 조작** → 페르소나 "최기준 점장"의 출근길 6:30 시나리오를 **모든 PR 검토의 기준선**으로 채택한다.

## 2. 핵심 토픽 1 — 모바일 최적화 API 응답 구조

### 2.1 PM 관점 요구사항

> 인용: 기획서 §3.4 직원 핵심 플로우 — *"전날 저녁 휴무 예정 푸시 알림"* 한 번 받고 앱을 켰을 때, **첫 화면이 4G 환경에서 1.5초 안에 그려져야** 직원이 이탈하지 않는다.

| # | 요구사항 | 측정 지표 |
|---|---|---|
| 1 | **첫 화면 TTI ≤ 1.5s** (4G, 미드레인지 안드로이드) | Performance trace 95p |
| 2 | **단일 진입 엔드포인트** (`/me/onboarding-status`)로 케이스 1~5 분기 (백엔드 Phase 26 완료) → 프론트는 부가 호출 없이 첫 화면 결정 | 첫 화면 진입까지 호출 수 ≤ 2 |
| 3 | **부분 갱신**: 휴무 수집 뷰에서 드래그 직후 재조회는 **충돌 검증 응답만** 받고 전체 페이로드 재로드 금지 | 평균 응답 ≤ 30KB |
| 4 | **페이지네이션 정책**: 스케줄 이력 / 근무 교환 이력은 cursor 기반. 첫 페이지 ≤ 20건 | 첫 응답 ≤ 50KB |
| 5 | **실패 시 fallback**: 네트워크 실패 시 AsyncStorage 마지막 캐시로 read-only 표시 | 오프라인 첫 화면 노출 100% |

### 2.2 백엔드에 던지는 PM 체크리스트

- [ ] `/me/schedules`(직원) 응답에 **이번 달 + 다음 달** 데이터를 한 번에 묶어달라 — 직원이 휴무 신청할 때 다음달이 항상 따라옴 (디자인 To Do "이번달/다음달 휴무 신청 분기" 참고)
- [ ] PNG 이미지 URL은 **CDN 절대 경로**로 응답 (RN `<Image>`에서 바로 prefetch 가능)
- [ ] 푸시 토큰 등록 API의 idempotency key 명시 (앱 재설치 / 재로그인 시 중복 등록 방지)
- [ ] 시간대(`Asia/Seoul`) 고정 정책을 응답 헤더 또는 contract에 명문화

## 3. 핵심 토픽 2 — RN 네이티브 연동 (푸시·AsyncStorage)

### 3.1 푸시 알림 시나리오 표

> 백엔드 메모: *"FCM 미연동 (NoOpAdapter)"* — 스케줄러 로직은 완성, **인프라 연결만 남음**.  
> 인용: 백엔드 §미결 사항 v1.2 이월

| 트리거 | 발송 시각 | 수신자 | 본문 예 | SLA | MVP? |
|---|---|---|---|---|---|
| 쉬기 전날 알림 | 매일 20:00 | 다음 날 휴무 직원 | "내일은 휴무예요. 푹 쉬세요." | ±5분 | **MUST** |
| 관리자 당일 근무자 | 매일 08:00 | 매장 관리자 | "오늘 근무자 3명 / 휴무 1명" | ±5분 | **MUST** |
| 휴무 재배치 결과 | 관리자 드래그 즉시 | 해당 직원 | "휴무가 6/17 → 6/19로 이동되었어요." | < 30초 | **MUST** |
| 근무 교환 요청·승인 | 이벤트 즉시 | 양측 | "박지호님이 6/22 교환을 요청했어요." | < 30초 | SHOULD |
| 미확인 스케줄 독촉 | 생성 후 24h 미확인 시 | 직원 | "이번 달 스케줄을 확인해주세요." | ±30분 | SHOULD |

### 3.2 AsyncStorage 데이터 카테고리 (PM 분류)

| 카테고리 | 항목 | 보관 기간 | 보안 등급 | 비고 |
|---|---|---|---|---|
| 인증 | Access Token (30분) | 메모리 + AsyncStorage 스냅샷 | **민감** | iOS Keychain / Android Keystore 권장 — 평문 AsyncStorage 금지 |
| 인증 | Refresh Token (7일, Rotation) | AsyncStorage(암호화) | **민감** | 위와 동일 |
| 사용자 | 내 프로필 (members + employees) | 무기한 (로그아웃까지) | 낮음 | 첫 화면 즉시 표시용 |
| 사용자 | 매장 정보 (storeName, primary 색상) | 무기한 | 낮음 | 케이스 3~5 환영 화면 즉시 표시 |
| 캐시 | 이번 달 / 다음 달 스케줄 | 7일 TTL | 낮음 | 오프라인 read-only |
| 큐 | 휴무 신청 오프라인 큐 | 송신 성공 시 제거 | 중간 | 지하철에서 신청 → 통신 복구 시 자동 전송 |
| 설정 | 푸시 알림 ON/OFF, 시간대 | 무기한 | 낮음 | 직원이 야간 근무자라면 20:00 알림 끌 수 있어야 |

### 3.3 PM 결정 사항

- [x] **JWT는 AsyncStorage 평문 저장 금지** — Keychain/Keystore 사용 (`react-native-keychain` 등)
- [x] **푸시 권한 요청 타이밍** = 카카오 OAuth 직후가 아니라 **첫 휴무 신청 완료 직후** (사용자가 가치를 본 다음에 요청해야 허용률이 높음)
- [x] **딥링크 미설치 케이스(Deferred Deep Link)** 토큰 소비 흐름은 SHOULD — 앱스토어 발급/도메인 발급 일정에 동기화

| KPI | 목표 | 측정 방법 | 데이터 소스 |
