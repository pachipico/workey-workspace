작성자: ux (최기준 점장 페르소나)
회의: frontend-kickoff-meeting
입력: ux-round1.md + ux-round2.md + pm·backend·frontend·design R1·R2 전부
관점: **매장 카운터에서 한 손으로 워키를 켜는 38세 점장**의 1인칭

> "두 라운드 동안 다들 자기 영역에서 합리적으로 응답해줬다. 절반은 합의됐고, 절반은 여전히 헛돌고 있다. 이번 R3에서 점장이 직접 도장을 찍는다. **양보 못 할 8개 항목, 양보 가능한 7개 항목, 그리고 미해결 5개 항목**으로 정리했다. 이 합의 그대로 v1.2 들어가지 않으면, 우리는 매장에서 다시 카톡으로 돌아간다. 이건 협박 아니라 사용자 사실이다."

## 0. 총평 — 두 라운드 끝나고 점장이 보는 풍경

### 잘된 부분 ✅
- **PM**이 R2에서 페르소나 정의를 PM(45세 / 06:30 지하철)에서 UX(38세 / 매장 카운터 30~90초)로 교체한 것은 **이번 회의 최대 성과**다. 페르소나가 흔들리면 모든 결정이 헛돈다.
- **Design**이 달력 today 색 sky-500 → sky-700 (8.75:1) 즉시 변경, BottomSheet 3단 30/60/100 확정, 매장 모드 고대비 토큰 신설 — UX R1·R2 핵심 요구 3건을 단호하게 받아줬다.
- **Frontend**가 R2에서 자기 R1을 자기가 수정(JWT MMKV → Keychain, FCM 토큰 엔드포인트 오류, 푸시 권한 타이밍)한 자세는 인정한다.
- **Backend**가 GET `/invite/{token}/preview` 신설 제안(PP-6 대응) + 2달 합산 `includeNext=true` 수용 — 점장 시각 발견을 API 스펙으로 옮긴 행동.

### 여전히 어긋난 부분 ❌
- **드래그 인터랙션 매핑 충돌** — frontend R2 §4-1: "long-press = 드래그 시작, short-press = 무동작" / PM R2 §2.2: "**반대로** — short-press = 어디로 옮길까요? 모드, long-press = 드래그". 점장 멘탈 모델은 PM 안에 가깝다. R3에서 결정.
- **PP-3(무료 티어 + 수동 추가 발견 어려움)이 어느 팀에서도 명시적 해결안 없음**. 백엔드는 "클라이언트 로직", 프론트는 코치마크 미언급, 디자인도 미언급.
- **Dark Mode** — PM은 v1.3 이월 주장 / Design은 라이트·다크 토큰 테이블 확정 / UX는 매장 모드 라이트 고대비 우선. 셋이 만나는 합의점이 명확치 않음.
- **TTI** — PM이 cold 1.5s / warm 0.8s 분리 안 제안. UX의 0.8s 요구는 warm으로만 충족됨. 점장 첫 진입(cold) 1.5s는 여전히 매장에서 길다.
- **CommonResponse unwrap** — backend가 frontend에 (A)/(B)/(C) 선택 묻고 frontend는 (A) 인터셉터 채택했는데, backend R2엔 확정 응답이 없음.

## 4. 4대 핵심 토픽 — UX 최종 결론

### T1. 모바일 최적화 API
- **MUST 채택**:
  - BFF `GET /me/home-summary` v1.2 (backend R2 §2-2 수용 / PM R2 §7 #12 MUST 등재). frontend는 client-side aggregation `Promise.all` 로 v1.1 임시 대응(frontend R2 §1-1) — 이중 안전장치.
  - ETag/If-None-Match → 스케줄 조회·연간 통계 v1.2 (backend R2 §6-2 Cache-Control 표 + frontend R2 §1-4 ETag 코드).
  - CommonResponse unwrap = **(A) Axios 인터셉터** (frontend R2 §1-2 채택, backend §질문 3 옵션 A) → backend R3에서 명시 확정 필요.
  - Asia/Seoul 응답 헤더 명문화 (backend R2 §1-5 수용).
- **양보**:
  - PNG CDN 절대 경로 → S3 인프라 후순위 (Y-2와 별개, MVP 외).
  - cursor 페이지네이션 → offset 유지 (backend R2 §1-6 거부 수용).
  - 스케줄 생성 비동기 + 푸시 결과 모델 → backend가 동기 10초 타임아웃 + design R2 §2 풀스크린 로딩 "약 3~5초 소요" 안내로 대체. **수용**. 단 직원 15명+ 매장은 95p 4초+ 가능 → v1.3 비동기 잡 검토 권고.
- **점장 시각 합격 기준**: 첫 화면 cold 1.5s / warm 0.8s 95p (Y-1). 매장 카운터에서 두 번째 진입 이후는 0.8s 보장.

  - **UX 권고: PM 안 채택**. 점장 멘탈 모델은 "탭 = 선택 / 길게 누름 = 메뉴/이동"이다. short-press를 선택 모드로 두면 한 손 시나리오의 일반 경로가 short-press만으로 완결된다. 드래그는 숙련 사용자 보조.
- **이월(v1.3)**:
  - 매장 모드 토글 UI (Y-4).
  - 좌·우손잡이 토글 (Y-5).
- **합격 시연**:
  - 베타 사용자 ≥ 15명 (UX R2 §1-2 표본 보강).
  - 두 시나리오 모두 NPS + 태스크 완수율 ≥ 85% (PM R2 §8 KPI).

## 5. design의 UX 검토 요청 2건 답변

### 검토 요청 1 (터치 타겟 현장 검증)
> "58dp가 충분한가, 60~64dp까지 확대해야 하는가? EmployeeChip 32pt가 첫 탭에 잡히는가?"

**점장 답변**:
- 라이트 모드 기본 48dp는 부족 → 매장 모드 58dp는 **충분**. 60~64dp는 화면 정보 밀도가 떨어져 6주 달력 그리드가 화면 밖으로 밀려난다. **58dp 유지 권고**.
- EmployeeChip 32pt는 매장 환경에서 **첫 탭 실패율 30%+** 추정. 기름·물·장갑 환경에서 손가락 정전식 인식이 불안정하기 때문. **44pt+ 표준 + 매장 모드 53pt+** 로 확정 (R-8).
- 추가 요구: 칩 외곽에 hitSlop `{top:6, bottom:6, left:6, right:6}` 적용 — 시각 너비 44pt 유지하면서 실 터치 영역 56pt 확보.

### 검토 요청 2 (야외/매장 시각 피로도)
> "sky-700 고채도가 매장 직사광에서 글레어 유발하지 않는가? 매장 모드를 라이트 고대비로 고정할지, 다크+고대비도 추가할지?"

**점장 답변**:
- 매장 카운터 시나리오는 **30~90초 자투리 사용**. 30분 이상 장시간 시각 피로도는 매장 시나리오 핵심 변수가 **아니다**.
- 30분 미만 단발 사용 시 라이트 고대비 > 다크 고대비 (직사광에서 다크 모드는 화면이 거울처럼 반사되어 더 안 보임).
- **결정 권고**: 매장 모드는 **라이트 고대비로 고정**. 다크 모드는 **직원 앱의 야간 시나리오** (저녁 8시 휴무 알림 받고 침대에서 확인) 전용으로 분리. v1.3에서 직원 앱에만 다크 모드 진입 UI 활성화.
- sky-700 글레어 우려는 **달력 today 배경에서만 8.75:1 충분**, 다른 표면(BottomSheet 핸들·CTA 버튼)은 sky-700 단독 면적이 작아 글레어 미미. 적용 OK.

## 8. 결론 — 점장 1인칭

> "두 라운드 동안 4팀이 정말 열심히 받아줬다. design은 대비 문제 즉시 인정·수정, frontend는 자기 R1을 자기가 고치는 자세, backend는 점장 시각을 API 신설로 받아주는 행동, PM은 페르소나를 바꾸는 결단 — 이런 모습이 사실상 v1.2 출시의 절반이다.
> 
> 그러나 출시까지 4개의 칼날이 남아 있다. **인터랙션 매핑(U-2), PP-3 코치마크(U-1), CommonResponse 응답(U-3), FCM 실연동 일정(U-4)** — 이 네 건이 R3에서 도장 안 찍히고 v1.2 들어가면, 매장 카운터에서 점장이 첫 진입 1분 안에 짜증을 내는 화면이 만들어진다.
> 
> 그리고 한 가지 더 — Red Line 8건 중 R-1·R-2·R-3·R-6은 **출시 차단 조건**으로 명문화해 달라. PM R2 §8 KPI 표에는 'TTI 95p ≤ 1.5s', '깔때기 완료율 ≥ 85%' 같은 측정 KPI는 있어도 'design Red Line 8건 PR 머지 가드' 같은 정성 차단 조건이 없다. v1.2 마지막 1주에 누군가 'sticky CTA 헤더로 임시 복귀'를 PR로 올리는 일이 없도록, **차단 룰을 코드/CI에 박아 두는 작업**이 필요하다.
> 
> 정리하자면, 매장 카운터에 한 손으로 폰 든 38세 점장이 30~90초 안에 카톡으로 도망치지 않게 만드는 일 — 이건 단일 팀이 못 한다. 4팀이 이번 회의에서 합의한 12건 + 미결 5건 결정 + Red Line 8건 차단 룰 = **이게 v1.2 점장 합격선의 전부다.** 다른 게 더 들어와도 좋지만, 이게 빠지면 다 무의미하다.
> 
> 카톡으로 돌아가지 않는 워키. 그 한 줄이 R3의 결론이다."
