## 0. 페르소나 충돌 — 회의의 전제부터 어긋났다

| 출처 | 페르소나 | 환경 | 시간 |
|---|---|---|---|
| **PM** (§5.1) | 최기준 점장 **45세, 강남 8평 카페, 직원 6명** | **출근길 지하철, 한 손에 아이스 아메리카노** | **06:30~06:45 (15분)** |
| **UX (R1 §0)** | 최기준 점장 **38세, 분식+카페 복합, 직원 8명(연동 4 / 미연동 4)** | **매장 카운터, 영업 중, 형광등 직사광, 70~80dB 소음** | **30~90초 자투리** |

> "PM이 그린 '지하철 출근길'은 비교적 안전한 환경이다. 한쪽 손에 음료수가 있다는 것 빼면 조명도 양호, 시간도 15분이나 있다. 우리(UX)가 그린 매장 카운터 시나리오가 훨씬 가혹하고, 실제 점장이 워키를 가장 자주 켜는 순간이다. **시나리오 두 개를 모두 합격 기준선으로 채택해야 한다.** 한쪽만 통과시키면 사용자 절반을 놓친다."

**요청 (전 팀)**: §1.1 시나리오 표준화 — UX의 "매장 모드(Store Mode)"를 **두 번째 합격 시나리오**로 §6 출시 합격 체크리스트에 추가 (PM에 후속 협의).

## 1. PM Round 1 반박

### 1-1. KPI "첫 화면 TTI ≤ 1.5s"는 너무 관대하다 (§6.1)
> "1.5초? 점장은 0.8초도 길다고 생각한다. 매장 카운터에서 손님 응대 사이 30초 자투리에 앱 켜는데, 1.5초 동안 흰 화면 보고 있으면 그냥 닫고 카톡 본다. **MMKV 캐시 기반 즉시 렌더(0.3s) + 백그라운드 갱신**을 표준으로 잡고, 1.5s는 cache miss 상한으로만 둬라."
- 위치: PM §6.1 KPI 표
- 요구: 두 종 KPI 분리 — `cold-start TTI ≤ 0.8s (캐시 hit, 95p)` / `cold-start TTI ≤ 1.5s (cache miss, 95p)` 두 개 모두 합격선으로.

### 1-2. NPS 7.0 합격선의 표본 부족 (§6.1·§6.2)
> "베타 사용자 5명 NPS 7.0? 5명에서 NPS 측정은 통계적으로 의미 없다. 한 명이 8점 한 명이 4점이면 평균은 6점이지만 그 4점이 진짜 시그널이다."
- 요구: 베타 표본 ≥ 15명, NPS 외에 **태스크 완수율(휴무 수집 뷰 진입 → 생성 확정 ≥ 85%)** 을 1차 합격선으로. NPS는 보조.

### 1-3. "출근길 06:30 시연" 단일 합격 시나리오의 함정 (§5.3)
> "지하철 시연만 통과하면 합격? 점심 피크 11–14시, 매장 형광등 + 외부 직사광 + 손이 젖은 상태 시연도 해야 한다. 지하철은 너무 친절한 환경이다."
- 요구: §5.3에 **시나리오 B (매장 영업 중 11~14시 점심 피크 시연)** 추가. UX가 직접 시연 진행.

### 1-4. "한 손 조작 모드 자동 감지 불가 → 옵션 토글 대신 기본값 한 손 친화" (§5.2)
> "기본값을 한 손 친화로 둔다는 건 좋다. 그러나 **좌손잡이는?** PM·디자인·프론트 모두 우측 FAB 전제. 좌손잡이는 우측 FAB가 Dead Zone이다. 한 손 모드 토글은 옵션이 아니라 **좌·우손 모드 토글**로 재정의해야 한다."
- 요구: 설정에 좌·우손잡이 토글 도입 (FAB·CTA 위치 좌우 반전).

### 1-5. "스케줄 생성 완료율 ≥ 85%" 측정 정의 모호 (§6.1)
> "휴무 수집 뷰 진입 → 확정의 전환율을 깔때기로 본다고 했는데, 점장이 도중에 닫고 다시 켜는 경우(매장 손님 응대로 인한 중단)가 많다. 그걸 이탈로 잡으면 합격 어렵다. **세션 단위가 아니라 일 단위 완수율**로 정의해야 한다."
- 요구: 24h 윈도우 내 동일 점장의 휴무 수집 뷰 진입 → 확정 도달 비율로 정의.

## 2. Backend Round 1 반박

### 2-1. BFF `/me/home-summary` v1.2 권장? — 점장 시각에선 v1.2 MUST다 (§2.1)
> "Promise.all 3개 병렬 호출로 우선 구현하고 v1.2에서 BFF 검토? 그 사이 점장은 매장 카운터에서 1.5초 흰 화면 보다가 앱 닫는다. **BFF 없이 1.5s TTI 95p 달성 불가**다. v1.2에서 BFF 결정을 미루지 말고 지금 잡아라."
- 위치: backend §2.1 마지막 줄 ("v1.1 범위 外")
- 요구: BFF 엔드포인트를 v1.2 MVP 합격선에 포함 — PM §1.2 우선순위 매트릭스에 추가 항목으로 등록.

### 2-2. 스케줄 생성 API "1~5초" 응답 시간 — 5초면 폰 던진다 (§2.3)
> "최악 5초? 그 5초 동안 점장은 화면 굳었다고 판단한다. 두 번 이상 같은 일이 일어나면 워키 끝났다."
- 위치: backend §2.3 표 ("1~5초 가능")
- 요구:
  1. 응답 시간 95p ≤ 2s SLO 약속.
  2. 그 이상 걸릴 가능성이 있으면 **비동기 잡 + 푸시 알림 결과** 모델로 전환 (`POST /schedules/generate` 즉시 202 + 완료 시 FCM data push). 모바일 한정 권장.
  3. 진행률 표시 — 클라이언트가 polling 또는 server-sent event 수신.

### 2-3. `CommonResponse.result` 항상 배열은 모바일에 독이다 (§4)
> "단건 조회도 `result[0]` 꺼내는 보일러플레이트가 RN 화면 진입마다 들어간다. 점장 측 부하는 0이지만, 프론트의 실수 한 번이 점장에게 빈 화면으로 돌아간다. **단건은 단건으로, 다건은 다건으로** 응답 분리하라."
- 요구: v1.2에서 `CommonResponse<T>` vs `CommonResponseList<T>` 분리 또는 `result`를 union으로 명시. 또는 RN 측에 표준 unwrap 유틸 강제 + lint rule.

### 2-4. ETag / Sparse fieldset "v1.2 검토" — 검토는 늦다 (§4)
> "모바일은 페이로드 100KB 차이가 LTE 환경에서 200~400ms 차이로 직접 보인다. 검토 단계에서 점장은 이미 카톡으로 돌아갔다."
- 요구:
  - **ETag**: 스케줄 조회·연간 통계에 우선 도입 (점장이 가장 자주 새로고침).
  - **Sparse fieldset**: 휴무 수집 뷰의 entries 응답에 `?fields=entries.date,entries.status` 적용.

### 2-5. "한 손 조작에 API 회수 2회면 무리 없다" 단언 부적절 (§2.4)
> "백엔드 입장에서 API 회수는 2회. 그러나 점장 손가락 입장에선 그 사이에 조건 입력 4필드 + 휴무 수집 뷰 드래그 + 확정 컨펌이 있다. **한 손 조작 평가는 백엔드 영역이 아니다.** 백엔드가 단언하면 잘못된 합의로 굳는다."
- 요구: backend §2.4를 "API 부담 평가"로 이름 변경. 한 손 조작 합격 여부는 UX 시연으로 판단 (R3 합의).

### 2-6. DEV 백도어 prd 전 삭제 — 잊으면 큰 사고
> "Phase 999가 prd 직전 단계라고 적혀 있다. 인간은 잊는다. **CI에 `/dev/**` 라우트 존재 시 prd 빌드 fail** 자동화 걸어라."
- 요구: 빌드 가드 자동화. PM에 후속 협의.

## 3. Frontend Round 1 반박

### 3-1. SplashScreen에서 onboarding-status 동기 호출 → 네트워크 단절 시 무한 로딩 (§2-1)
> "지하철 음영, 매장 와이파이 끊김. SplashScreen이 `await api.getOnboardingStatus()` 기다리면 점장 폰은 splash에서 멈춘다. 점장은 강제 종료 후 다시 켠다. 무한 루프."
- 위치: frontend §2-1 SplashScreen 코드
- 요구: **MMKV에 마지막 onboarding-status 캐시** → 캐시 hit 시 즉시 라우팅 + 백그라운드 재검증. cache miss + 네트워크 실패면 "오프라인 모드로 마지막 화면 보기" 선택지 제공.

### 3-2. BottomSheet snap points `['35%']` 단일 — 충돌 바텀시트엔 부족 (§3-1)
> "InviteBottomSheet 35%는 OK. 하지만 충돌 해소 시트는 신청자 칩이 3~5명 + 안내 문구 + 드롭 힌트가 들어가야 한다. 35%는 칩 3명까지 보고 그 이상은 스크롤. 점장은 한 손으로 그 안에서 또 스크롤하다 짜증."
- 위치: frontend §3-1 + 디자인 §2.1.1
- 요구: BottomSheet 종류별 snap points 표준 정의 — `Invite: ['40%']`, `LeaveRequest: ['50%']`, `DayOffConflict: ['35%', '60%', '90%']`, `ShiftSwap: ['60%']`. 디자인 측에 토큰화 요청 (R2 design 질문).

### 3-3. 드래그&드롭 fallback의 mental model 부재 (§4-3)
> "longPress = 드래그, 짧은 탭 = 날짜 선택? 점장은 그 차이를 모른다. **첫 진입 시 1회 코치마크** 또는 **칩에 명시적 'Move' 아이콘 + 메뉴**가 필요하다."
- 위치: frontend §4-3 DayOffChip 코드
- 요구:
  - 칩 우측에 ⋮ 아이콘 → 탭 시 액션시트("이 날짜로 이동" 셀 직접 선택 / 드래그로 이동).
  - 휴무 수집 뷰 첫 진입 시 코치마크 1회 ("칩을 길게 눌러 끌거나, ⋮ 메뉴에서 이동을 선택하세요").

### 3-4. 오프라인 큐 적용 범위가 좁다 (§5-2)
> "휴무 신청·근무 교환만 큐에 들어간다고 했다. 그런데 **휴무 수집 뷰의 드래그 결과(재배치)** 도 영업 중 자주 발생하는데 큐 미적용. 와이파이 끊김에서 드래그 → 실패 → 다시 드래그 → 또 실패 = 점장 폭발."
- 요구: `PATCH /day-off-requests/{id}/relocate` (또는 백엔드 응답 endpoint) 도 오프라인 큐 대상으로 추가. 큐 진입 시 사용자에게 토스트 ("저장 대기 중 — 통신 복구 시 자동 전송").

### 3-5. DiceBear SVG 30종 — 무료티어 진입 부담 (§7·§8)
> "무료티어 5명 매장 점장이 워키 처음 켤 때 30종 SVG 다 받는 건 과하다. 첫 다운로드 무거우면 설치 자체를 포기한다."
- 위치: frontend §8 리스크 표
- 요구: SVG 사이드로드 (lazy fetch) — 첫 로딩에는 기본 4~6종만 번들, 나머지는 직원 등록 화면 진입 시점에 prefetch.

### 3-6. "기본값을 한 손 친화로 설계" — 우손잡이 전제만 있음 (§4-1)
> "FAB right: 24, bottom: 24 — 우손잡이 전제다. 좌손잡이 점장은 매장에서 늘 우측 손에 카드 단말기 + 음료를 든다. **좌·우 토글이 옵션이 아니라 필수.**"
- 요구: 설정에 좌·우손 토글 + 기본값은 시스템 손 선택 (iOS 16+ / Android는 사용자 입력) 또는 첫 온보딩에서 묻기.

### 3-7. React Navigation v7 + Reanimated 3 + gesture-handler 안정성 (§7)
> "React Navigation v7은 안정 stable 진입했지만, Reanimated 3 + gesture-handler 5의 worklet 패턴은 매장 환경 미드레인지 안드로이드(예: 갤럭시 A 시리즈, 4GB RAM)에서 60fps 보장이 어렵다. 데모 디바이스 외에 **저사양 기기 1종 베타 합격선** 추가."
- 요구: 베타 디바이스 매트릭스에 갤럭시 A 시리즈 또는 동급 1종 포함 (예: A24/A34, RAM 4GB).

## 4. Design Round 1 반박

### 4-1. 달력 오늘 날짜 배경 #0EA5E9 + 흰 텍스트 2.9:1 — 매장 직사광에서 못 읽음 (§2.4)
> "디자인이 직접 '⚠️ 흰 텍스트 볼드 처리 또는 #0369A1 배경으로 대체 고려'라고 적었다. **'고려'가 아니라 결정**해라. 매장 직사광에서 2.9:1은 읽을 수 없다. 즉시 sky-700(#0369A1)로 바꿔야 한다."
- 위치: design §2.4 마지막 행
- 요구: 달력 오늘 날짜 배경 토큰을 sky-700로 즉시 변경. `--ts-cal-today-bg` 갱신.

### 4-2. Dark Mode 미정의 + "고조도 환경 대응" 표는 있는데 매장 모드 토글이 없다 (§2.4·§5)
> "고조도 환경 대응 표를 만들어 놓고 정작 사용자가 토글할 수단을 안 줬다. UX R1에서 이미 'Store Mode' 토글을 제안했는데 design R1엔 응답이 없다."
- 요구: design §2.4에 **High-Contrast(매장 모드) 토큰 변형 정의** 추가. 라이트·다크·매장 모드 3종 팔레트.

### 4-3. BottomSheet 핸들 너비 32pt 색상 sky-300 — 매장 직사광 가시성 부족 (§2.1.1)
> "32pt × 4pt + sky-300(#7DD3FC)? 직사광에서 sky-50 배경 위 sky-300 핸들은 거의 안 보인다. **너비 ≥ 40dp + 색상 sky-500 이상**으로 보강해야 점장이 '여기를 잡으면 닫힌다'를 직관 인식한다."
- 요구: BottomSheet 핸들 너비 40~48dp / 색상 `--ts-primary-ink(sky-700)` 변형 토큰.

### 4-4. Tab Bar 활성/비활성을 색만으로 구분 (§2.1.3)
> "활성 sky-500 / 비활성 slate-400 — 색약 + 노안 + 직사광 = 식별 불가. **아이콘 형태(채워짐 vs 외곽선)**도 함께 변경해야 한다."
- 요구: design 시스템에 active/inactive 아이콘 페어 정의. (아이콘 SVG 미전환 항목과 묶어 처리)

### 4-5. 헤더 우측 "생성 확정" 버튼 → 하단 sticky 권고는 옳다 (§2.3)
> "design이 권고한 부분은 UX와 100% 일치한다. PM과 frontend도 동일 채택해야. **R3에서 만장일치 항목으로 굳히자.**"
- 요구 (전 팀): R3 합의 시 §2.3 권고를 디자인 시스템 표준으로 명문화.

### 4-6. EmployeeChip minHeight 32pt → 점장 시각 부족 (§2.2)
> "직원 칩 minHeight 32pt가 '드래그 소스'로는 작다. 노안 점장이 이 칩을 잡고 끌어야 하는데 32pt는 손가락에 다 가린다. **44pt 이상**으로 키워야 매장 환경에서 잡힌다."
- 요구: EmployeeChip minHeight 44pt 표준화. 가로는 가변 OK.

## 5. 4대 핵심 토픽 — 점장 시각 합의 후보

## 7. 결론 (점장 1인칭)

> "PM은 지하철 시나리오로, 백엔드는 API 회수로, 프론트는 라이브러리 안정성으로, 디자인은 토큰 일관성으로 — 다들 자기 영역에서 합리적이다. 하지만 매장 카운터의 형광등 직사광 아래 한 손으로 워키를 켠 38세 점장에겐 **전 팀의 결정이 합쳐져서 한 화면으로 도착**한다. R3에선 §5의 합의 후보 21건 중 최소 15건을 합의로 굳히고, 합의 안 된 것은 명시적 미해결 리스트로 남겨라. 모호한 채로 v1.2 들어가면 매장에서 '이 앱은 안 된다'는 인상 한 번 박힌 뒤엔 회복 어렵다."
